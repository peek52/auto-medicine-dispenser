1Boot Sequence - ลำดับการเริ่มทำงาน

ตั้งแต่ app_main() จนถึง init complete ใช้เวลาประมาณ 13-18 วินาที

POWERON / RESET Bootloader (ESP-ROM) -> load app from flash @ 80 MHz PSRAM init @ 200 MHz (10x faster than default 20 MHz) CPU clock 360 MHz dual-core RISC-V | v app_main() phase (~0-18 s) 1. NVS init 2. cloud_secrets load (Telegram + Sheets tokens) 3. I2C bus init SDA=7 SCL=8 @ 50 kHz 4. Probe TCA9548A -> PCA9685 -> PCF8574 (each retry x N) 5. FT6336U init (3x retry + 4x ping fallback) 6. VL53L0X bootstrap (6 ch via TCA mux) 7. Display ST7796S init (SPI 40 MHz) 8. Clock task starts 9. Liveness canary task (TWDT 30 s) 10. deferred_init_task spawned | v deferred_init_task (~18-30 s) - dfplayer_init (UART audio) - settings_load_nvs (volume / language / config) - start_webserver (port 80, /tech endpoints) - start_stream_server (port 81, MJPEG) - dispenser_scheduler_start (alarm task) - usb_mouse_start - wifi_sta_init (esp_hosted SDIO co-processor, ~7 s) - SNTP sync time - netpie_init (MQTT broker) - telegram_polling_task starts

เงื่อนไขการเข้าสู่ "Safe mode"

  • REMOVED Safe mode ไม่ทำงานแล้ว (user request) - แม้ panic ครั้งก่อน ระบบเข้า full stack เสมอ
  • ทุกอย่างยังเปิด หลัง panic: NTP sync, WiFi, camera, VL53, Telegram
  • เพียง s_skip_i2c_restart = true ถ้า reset 3 ครั้งติด - ผู้ใช้ต้อง power-cycle

เงื่อนไข "All I2C dead - disabling master bus"

  • หาก probe ที่ boot phase: PCA9685 + PCF8574 + FT6336U ทั้งหมด NACK
  • -> i2c_manager_disable_bus() permanent ของ boot นี้ (ป้องกัน i2c_master ISR race panic)
  • Display + WiFi ยังทำงาน, แต่ touch/servo/IR/VL53 ปิด - ต้อง power-cycle

2VL53L0X - กลไก Init + Recovery

เซ็นเซอร์ ToF 6 ตัว ผ่าน TCA9548A (channel 0-5) อ่านระยะยาในกระบอก คำนวณ pill_count

Init Sequence ต่อช่อง

1
vl53_release_xshut() - set ทุก XSHUT pin (GPIO 20/22/23/47/48/53) เป็น HIGH
2
tca9548a_disable_all() - ปิดทุก channel ก่อน -> vTaskDelay(250 ms) ให้ chip boot
3
For each channel 0-5: tca_select -> vl53_wait_for_model_id -> vl53_init_device (~70 register ops) -> start_continuous
4
ถ้า fail: XSHUT toggle hard-reset (LOW 5 ms -> HIGH 30 ms) -> retry init (สูงสุด 3 ครั้ง)

Per-op Retry (ใน vl53_io)

  • 5-attempt exponential backoff: 0 / 200 us / 1 ms / 3 ms / 8 ms
  • Re-select TCA channel ทุก retry (กัน channel-byte หาย)
  • แก้ปัญหา IDF v5.3.2 i2c_master ISR race: write หลัง read return INVALID_STATE

Background poll task

เงื่อนไขการทำงาน
Idlepoll ทุก 5 วินาที (อ่านระยะ, คำนวณ pill_count)
Camera busypause poll (cooperative) ระหว่าง SCCB burst
Sensor missingretry probe ทุก 5 นาที (สูงสุด 5 ครั้ง -> permanently_missing)
หลัง dispense / count adjustvl53l0x_request_refresh() -> poll ทันทีรอบเดียว
Fix วันนี้ (2026-05-06): Late-detect bug - TCA ping หลัง VL53 init storm NACK ชั่วคราว -> re-bootstrap loop -> ใช้ tca_was_up_at_boot flag แทน

3Touch (FT6336U) - Auto-Recovery

หน้าจอสัมผัส capacitive 480x320, อ่านผ่าน I2C @ addr 0x38

Read flow

1
clock_task เรียก ft6336u_read_touch() ใน main loop
2
อ่าน register 0x02 (6 bytes - touch count + x + y + weight)
3
Cache: idle 20 ms (50 Hz), pressed 10 ms (100 Hz)
4
PRESS_DEBOUNCE = 2, COORD_STABILITY 150 px, MIN_REPRESS_GAP 150 ms, BOOT_MUTE 3 s

เงื่อนไขกู้คืนเมื่อ I2C ค้าง

ConditionAction
fail 8 ครั้งติด (~160 ms)i2c_manager_recover_bus() bit-bang SDA + recreate master bus
recovery cool-down30 วินาที ระหว่าง attempt (กัน loop)
หลัง recoveryforce s_touch_initialized = false -> re-init รอบถัดไป
chip-id NACK ที่ bootลอง 3 รอบ + 4x ping -> fallback mode (ใช้ register read แม้ ID อ่านไม่ได้)
Fix วันนี้: Auto-recovery เพิ่มมาวันนี้ - เดิม touch fail แล้ว stuck ตลอดจนกว่า power-cycle ปัจจุบันกู้คืนเองภายใน ~200 ms

4Scheduled Dispense - จ่ายยาตามเวลา

Trigger conditions

  • RTC ถึงเวลาที่ตั้งใน schedule (เช่น 08:00, 12:00, 18:00)
  • med ที่ count > 0 และ enable = 1 ใน slot นั้น
  • g_quiet_hours_disabled = false หรืออยู่นอก quiet window
  • System NTP synced (ถ้าไม่ sync -> ข้ามจนกว่าจะ sync)

Per-pill flow (ต่อตัวยาต่อ slot)

1
dispenser_set_busy() + พูดเสียง track 32 (กรุณารอ)
2
PCF8574 warmup x 8 reads (10 ms gap) - ล้าง stale state
3
servo go_work (ดันยาออก) -> loop start, IR poll ~100 Hz
4
หลัง 1.5 s -> servo go_home (ดึงกลับ)
5
IR poll ต่ออีก 4 + 2 s = 7.5 s รวม. ตรวจ 3 consecutive LOW reads = pill detected
6
ถ้า detected -> ลด count ใน NETPIE shadow + audit log + send Telegram photo
7
ถ้า miss -> append name ลง missed_meds[] + เสียง track 84 (ไม่พบยา)

เงื่อนไขเสียงประกาศ

สถานการณ์เสียง
เริ่มจ่ายยาtrack 28 + 32
จ่ายยาสำเร็จ >=1 เม็ดtrack 9 (โปรดทานยา)
ไม่พบยาเลยtrack 7 (no meds)
ใกล้หมด <=2 เม็ดtrack 22 (ใกล้หมด)
ระบบหยุดเพราะ I2C errortrack 84
Fix วันนี้: IR phantom-detection จาก servo PWM coupling - เพิ่ม IR_DEBOUNCE_LOW_SAMPLES = 3 (3 ครั้งติด LOW = ตรวจพบจริง) เดิม single read = ตรวจพบ -> spike ของ servo PWM ถูกนับเป็นยา

5Manual Dispense - คืนยา / +/-/ All

Trigger sources

  • UI หน้าเครื่อง: ปุ่ม "+" / "-" ในหน้า meds setup
  • UI หน้าเครื่อง: ปุ่ม "คืนยาทั้งหมด" (qty=100 -> ALL mode)
  • Web /tech: POST /tech/dispense?med=N&qty=N

เงื่อนไข qty

qtyความหมาย
1-99จ่ายตามจำนวน
100ALL mode - eject ทั้งหมด ใช้ IR เป็น STOP signal (cartridge หมด)

เงื่อนไขหยุด ALL mode

  • EMPTY_CONFIRM_CYCLES = 1 -> ถ้าไม่พบยา 1 รอบติด -> หยุด
  • กรณี IR ปิด (servo-trust mode) -> loop จนถึง MAX_PILLS แล้วหยุด
  • หาก err1 != ESP_OK (servo command fail) -> bail ทันที + Telegram fail message

Servo positions

Channelhome (degrees)work (degrees)หมายเหตุ
1-666 (default)33 (default)ปรับได้ผ่าน /tech/servo, NVS-saved

Camera lazy init

1
First call to camera_ensure_initialized()
2
vl53_pause() + 250 ms wait (กัน I2C contention)
3
JPEG encoder init (out_buf 1 MB x 2 + rot_buf 1 MB ใน PSRAM)
4
AVDD power-cycle: acquire -> release -> 200 ms -> re-acquire (กัน inrush latch)
5
SCCB detect OV5647 (PID 0x5647) -> format set -> stream-on
6
Camera task starts -> 700 ms warmup ก่อนรูปแรกพร้อม
7
vl53_resume() + กลับมา poll ปกติ

8Network - WiFi / NTP / NETPIE / Offline

เชื่อมต่อลำดับ

1
esp_hosted SDIO reset slave (GPIO 54) -> init transport via SDIO 4-bit @ 40 MHz
2
WiFi STA connect -> SSID/PASS จาก NVS (default: PJ_Main router_true)
3
SNTP sync -> RTC DS3231 update
4
NETPIE MQTT broker (mqtt.netpie.io:1883) - shadow sync
5
Telegram polling task starts

Offline event queue

  • หาก Telegram/HTTP fail -> offline_sync_queue_event() เก็บ event ลง NVS
  • Background retry ทุก 60 วินาที - ส่งซ้ำเมื่อ network กลับมา
  • Event types: 1=DOSE, 2=MISS, 3=PHOTO, 4=ADJUST
