## 2.2.3 ขาอินพุต/เอาต์พุต (I/O Peripherals)

ไมโครคอนโทรลเลอร์ ESP32-P4 มีจำนวนขาสัญญาณดิจิตอลอเนกประสงค์ (General-Purpose Input/Output: GPIO) รวมทั้งสิ้น 55 ขา ซึ่งครอบคลุมการรองรับโปรโตคอลและฟังก์ชันต่าง ๆ ได้อย่างหลากหลาย แต่ในรูปแบบของ development board "ESP32-P4-Nano" ของบริษัท Waveshare ที่ใช้ในโครงงานนี้ ผู้ผลิตได้นำขา GPIO ออกมาให้เข้าถึงผ่าน pin header สีเหลือง 28 ขาเท่านั้น เนื่องจากขาที่เหลือถูกสำรองไว้สำหรับฟังก์ชันเฉพาะ ได้แก่ การเชื่อมต่อกับชิปร่วม ESP32-C6 (สำหรับ WiFi/Bluetooth) ผ่าน SDIO interface, การเชื่อมต่อกับ PSRAM ภายใน, การเชื่อมต่อกล้อง MIPI-CSI 2 lanes, และการเชื่อมต่อจอ MIPI-DSI สำรอง

ขาที่นำออกมาบน header ของบอร์ด ESP32-P4-Nano ในโครงงานนี้ถูกใช้งานครอบคลุมตามฟังก์ชันต่าง ๆ ดังนี้

**บัส I²C (Inter-Integrated Circuit) จำนวน 2 ชุด** ค่าเริ่มต้นจากผู้ผลิตคือ I²C0 ที่ใช้ขา SCL บน GPIO8 และ SDA บน GPIO7 ซึ่งโครงงานนี้ใช้เป็นบัสร่วมสำหรับเชื่อมต่ออุปกรณ์ I²C ทั้งหมด 7 ชิ้น ได้แก่ TCA9548A (ที่อยู่ 0x70), PCA9685 (0x40), PCF8574 (0x20), DS3231 (0x68), FT6336U (0x38), OV5647 SCCB (0x36) และ AT24Cxx EEPROM บน RTC module (0x56) บัส I²C1 ถูกสำรองไว้สำหรับการขยายในอนาคต

**บัส SPI (Serial Peripheral Interface) จำนวน 5 ชุด** SPI2 ทำงานความเร็ว 40 MHz ใช้สำหรับขับจอแสดงผล ST7796S TFT ขนาด 480×320 พิกเซล โดยใช้ขา MOSI=GPIO32, SCK=GPIO36, CS=GPIO26, DC=GPIO24 และ RST=GPIO25 ส่วน SPI ตัวอื่น ๆ สำรองไว้สำหรับงานเสริมเช่นการเชื่อมต่อ flash ภายนอกหรือ ADC ความเร็วสูง

**UART (Universal Asynchronous Receiver/Transmitter) จำนวน 6 ชุด** UART0 ถูกใช้เป็น console serial port มาตรฐานสำหรับการ flash และ debug, UART1 ถูกใช้เชื่อมต่อกับโมดูลขยายเสียง DY-HV20T ที่ความเร็ว baud rate 9600 bps โดยใช้ขา TX=GPIO37 และ RX=GPIO38

**SDIO (Secure Digital Input/Output)** ใช้เชื่อมต่อกับชิปร่วม ESP32-C6 ซึ่งทำหน้าที่เป็น WiFi co-processor ผ่าน 4-bit data bus ที่ความเร็ว 40 MHz ใช้ขา GPIO14–GPIO19 รวม 6 ขา (CMD, CLK, D0–D3) การออกแบบนี้ทำให้ ESP32-P4 ไม่ต้องประมวลผล WiFi stack ภายในตัวเอง สามารถมุ่งเน้นทรัพยากรไปที่งานควบคุมเซนเซอร์และประมวลผลภาพ

**MIPI-CSI (Camera Serial Interface)** เชื่อมต่อกับเซนเซอร์ภาพ OV5647 ผ่าน 2 data lanes บวก 1 clock lane รวม 6 เส้น differential pair รองรับอัตราการส่งข้อมูลสูงสุด 1.5 Gbps ต่อ lane ข้อมูล Bayer raw 8-bit เคลื่อนที่ผ่านบัสนี้เข้าสู่ ISP ภายในชิปโดยตรง ไม่ผ่าน CPU ทำให้ลดภาระการประมวลผลของแกน RISC-V

**LEDC (LED Controller) PWM Channel** ใช้สำหรับสร้างสัญญาณนาฬิกา 24 MHz บน GPIO33 ซึ่งทำหน้าที่เป็น XCLK input ของกล้อง OV5647 เพื่อให้ PLL ภายในกล้องทำงานได้ตามสเปก

**GPIO Output ทั่วไป** ใช้ควบคุม XSHUT line ของเซนเซอร์ VL53L0X ทั้ง 6 ตัวบน GPIO20, 22, 23, 47, 48 และ 53 รวมถึง GPIO21 ที่เชื่อมต่อกับขา reset ของ FT6336U เพื่อใช้ฮาร์ดรีเซ็ตชิปสัมผัสเมื่อเกิดความผิดพลาด

## 2.3 ESP-IDF Framework และ FreeRTOS

### 2.3.1 ESP-IDF Framework และโครงสร้างคอมโพเนนต์

ESP-IDF (Espressif IoT Development Framework) คือเฟรมเวิร์กที่บริษัท Espressif Systems พัฒนาขึ้นเพื่อใช้พัฒนาซอฟต์แวร์บนชิป ESP32 ทุกรุ่น เวอร์ชันที่ใช้ในโครงงานนี้คือ v5.3.2 ซึ่งเป็นเวอร์ชันที่รองรับ ESP32-P4 อย่างเต็มรูปแบบ โดยเฟรมเวิร์กออกแบบเป็นสถาปัตยกรรมแบบ component-based ที่อาศัยระบบ build CMake ผู้พัฒนาสามารถนำคอมโพเนนต์ที่จัดเตรียมไว้แล้วมาใช้งานได้ทันที เช่น `driver/i2c_master`, `esp_driver_cam`, `esp_driver_isp`, `esp_driver_jpeg`, `esp_wifi`, `mqtt_client`, `esp_http_client`, `nvs_flash` และอีกกว่า 100 คอมโพเนนต์ภายในระบบ

โครงสร้างของแต่ละคอมโพเนนต์ในการพัฒนาแบ่งเป็น 4 ชั้นหลัก ได้แก่

**Application Layer (ชั้นโปรแกรมประยุกต์)** — โค้ดที่ผู้พัฒนาเขียนขึ้นเองทั้งหมด เช่น `main.c`, `dispenser_scheduler.c`, `ui_standby.cpp` ในโครงงานนี้

**Driver Layer (ชั้นไดรเวอร์ระดับสูง)** — API สาธารณะที่เปิดให้ผู้พัฒนาเรียกใช้ เช่น `i2c_master_transmit()`, `esp_cam_sensor_set_format()` ที่ปกปิดรายละเอียดของฮาร์ดแวร์เอาไว้

**HAL Layer (Hardware Abstraction Layer)** — ชั้นที่นามธรรม register-level operations ออกมาเป็นฟังก์ชันที่อ่านง่าย เช่น `i2c_master_hal_send_start()` หรือ `isp_hal_set_demosaic_format()` ทำให้ ESP-IDF เดียวกันรองรับ ESP32 รุ่นต่าง ๆ ได้

**LL Layer (Low-Level Layer)** — เขียน register โดยตรงเป็น static inline function เพื่อความเร็ว ไม่ใช้ resource มาก เช่น `isp_ll_clk_enable()` ที่กำหนดบิตในรีจิสเตอร์ภายในชิปได้ทันที

นอกจากคอมโพเนนต์มาตรฐานแล้ว ESP-IDF ยังรองรับระบบ "Managed Components" ผ่าน Espressif Component Registry (registry.idf.com) ซึ่งโครงงานนี้ดึงคอมโพเนนต์ภายนอก 3 ตัว ได้แก่ `espressif/esp_cam_sensor` (สำหรับไดรเวอร์ OV5647), `espressif/esp_sccb_intf` (สำหรับโปรโตคอล SCCB) และ `espressif/esp_hosted` (สำหรับ WiFi co-processor บน ESP32-C6)

### 2.3.2 ระบบปฏิบัติการเวลาจริง FreeRTOS

FreeRTOS เป็นเคอร์เนลของระบบปฏิบัติการเวลาจริง (Real-Time Operating System: RTOS) ที่บรรจุมาพร้อมกับ ESP-IDF เป็นค่าเริ่มต้น และได้รับการดัดแปลงเล็กน้อยเพื่อรองรับสถาปัตยกรรม dual-core ของ ESP32-P4 หน้าที่หลักของ FreeRTOS ในโครงงานนี้คือการจัดการการทำงานหลายภาระงาน (multitasking) อย่างเป็นระบบ โดยอาศัยตัวจัดตารางการทำงานที่ใช้นโยบาย Preemptive Priority-based Scheduling

ในโมเดลนี้ แต่ละ task มีระดับความสำคัญเป็นตัวเลข 0–25 (โดย 0 คือ idle task และ 25 คือ critical task) เมื่อมี task ที่มีความสำคัญสูงกว่าพร้อมทำงาน เคอร์เนลจะหยุด task ปัจจุบันที่กำลังทำงานอยู่ทันที (preempt) และเปลี่ยนไปประมวลผล task ใหม่ ทำให้การตอบสนองต่อ event สำคัญ เช่น การกดหน้าจอสัมผัส หรือการตรวจจับเม็ดยาผ่าน IR sensor เกิดขึ้นในช่วงเวลาที่คาดเดาได้ (deterministic timing)

โครงงานนี้ใช้กลไก synchronization ของ FreeRTOS หลายรูปแบบ ได้แก่ **Mutex** สำหรับป้องกันการเข้าใช้ทรัพยากรร่วมพร้อมกัน เช่น `g_i2c_mutex` ที่ปกป้องบัส I²C, **Binary Semaphore** สำหรับการ signal ระหว่าง ISR กับ task เช่น `frame_ready_sem` ที่ ISR ของกล้องใช้แจ้ง task การประมวลผล, **Queue** สำหรับการส่งข้อมูลแบบ FIFO ระหว่าง task เช่น Telegram text queue ที่บัฟเฟอร์ข้อความ 16 รายการก่อนส่งออก HTTPS, และ **Event Group** สำหรับการรอเงื่อนไขหลายอย่างพร้อมกัน เช่น `wifi_event_group` ที่ task หลักรอบิตของ `WIFI_CONNECTED_BIT` หรือ `WIFI_FAIL_BIT`

### 2.3.3 ตัวจับเวลาเฝ้าระวัง (Watchdog Timers)

ESP32-P4 มี watchdog timer สามชนิดที่ทำงานควบคู่กันเพื่อรับประกันความเสถียรของระบบ ได้แก่

**Task Watchdog Timer (TWDT)** เป็น software watchdog ที่ตรวจสอบว่า task ที่ subscribe ไว้ได้ "feed" watchdog ภายในระยะเวลาที่กำหนดหรือไม่ โครงงานนี้ตั้งค่า TWDT timeout ไว้ที่ 30 วินาที และ subscribe IDLE0 / IDLE1 task ของ FreeRTOS รวมถึงสร้าง task เฉพาะชื่อ `liveness_canary_task` ที่ feed TWDT ทุก 2 วินาที หาก scheduler ค้างเกิน 30 วินาที (เช่นจาก deadlock หรือ priority inversion) TWDT จะ trigger panic handler ที่นำระบบ reset อัตโนมัติ

**Interrupt Watchdog Timer (INT_WDT)** ตรวจจับกรณีที่ interrupt service routine (ISR) ทำงานนานเกินไปและบล็อก ISR อื่น ๆ ตั้งค่า timeout ที่ 300 มิลลิวินาที ซึ่งเป็นการรับประกันว่าจะไม่เกิดสถานการณ์ที่ ISR ค้างนานจนทำให้ระบบไม่ตอบสนอง

**RTC Watchdog (RTC_WDT)** เป็น hardware watchdog ที่ทำงานในระดับชิป ไม่ขึ้นกับสถานะของซอฟต์แวร์ใด ๆ ใช้สำหรับกรณีฉุกเฉินที่สุด เช่น CPU หยุดทำงานทั้งหมด หรือเกิด data corruption ใน memory หาก RTC_WDT ไม่ได้รับการ feed ภายในเวลาที่กำหนด ระบบจะถูกรีเซ็ตในระดับฮาร์ดแวร์เสมือนการกดปุ่ม EN

## 2.4 โปรโตคอลการสื่อสารฮาร์ดแวร์

### 2.4.1 โปรโตคอล I²C (Inter-Integrated Circuit)

I²C เป็นบัสสื่อสารแบบสองสาย (Two-Wire Serial Bus) ที่พัฒนาโดยบริษัท Philips Semiconductors (ปัจจุบันคือ NXP) ในปี ค.ศ. 1982 ประกอบด้วยสายสัญญาณเพียง 2 เส้น ได้แก่ Serial Data (SDA) และ Serial Clock (SCL) ทั้งสองเส้นต่อกับแหล่งจ่าย VCC ผ่านตัวต้านทาน Pull-up resistor บัสนี้ทำงานในรูปแบบ master-slave โดยมี master ที่ทำหน้าที่กำเนิดสัญญาณ clock และเริ่มต้นการสื่อสาร ในขณะที่ slave จะตอบสนองตาม address ที่ master ระบุ

อุปกรณ์ทุกตัวบนบัส I²C ต้องมี address เฉพาะตัวขนาด 7 บิต (รองรับสูงสุด 128 อุปกรณ์ในทางทฤษฎี แต่ในทางปฏิบัติ address บางส่วนถูกสำรองไว้) การสื่อสารหนึ่งครั้ง (transaction) ประกอบด้วยขั้นตอน START condition → 7-bit address + R/W bit → ACK → ส่งหรือรับ data byte → ACK → ... → STOP condition

**การคำนวณ Pull-up Resistor** ค่า resistor ที่เหมาะสมต้องสมดุลระหว่างสองข้อจำกัด ได้แก่ การดึงสัญญาณให้ลงไปที่ระดับ logic LOW ภายในเวลาที่ slave คาดหวัง (typically VOL ≤ 0.4 V ที่กระแส 3 mA) ตามสมการ R_max = (VCC - VOL) / I_OL = (3.3 - 0.4) / 0.003 = 967 Ω และการประจุ-คายประจุ capacitance ของบัสภายในเวลาที่ rising edge ของ clock cycle อนุญาต ตามสมการ R_min = t_r / (0.847 × C_b) สำหรับโครงงานนี้ที่ใช้ความถี่ 50 kHz (1/50000 = 20 µs ต่อรอบ) และ bus capacitance ประมาณ 200 pF จะคำนวณได้ R_min ≈ 5.9 kΩ ในทางปฏิบัติเลือกใช้ตัวต้านทาน 4.7 kΩ ซึ่งอยู่ในช่วงที่ครอบคลุม

**โหมดความเร็วของ I²C** ตาม spec แบ่งเป็น 4 ระดับ ได้แก่ Standard Mode 100 kHz, Fast Mode 400 kHz, Fast-mode Plus 1 MHz และ High-speed Mode 3.4 MHz โครงงานนี้เลือกใช้ความเร็วที่ต่ำกว่ามาตรฐาน คือ 50 kHz เพื่อรองรับสายเชื่อมต่อยาวระหว่างบอร์ดควบคุมและโมดูล VL53L0X 6 ตัว ที่อยู่ห่างจากบอร์ดประมาณ 30–40 เซนติเมตร ความยาวสายดังกล่าวเพิ่ม bus capacitance สูงขึ้นกว่าค่าปกติ การลดความเร็วช่วยให้ rising edge มีเวลาเพียงพอจะกลับสู่ logic HIGH ก่อนจะถึง sampling edge ของ clock ถัดไป

### 2.4.2 I²C Multiplexer (TCA9548A)

ปัญหาสำคัญในโครงงานนี้คือเซนเซอร์ VL53L0X ทุกตัวมี I²C address เริ่มต้นเป็น 0x29 ซึ่งทำให้ไม่สามารถวางทั้ง 6 ตัวบนบัสเดียวกันได้โดยตรง การแก้ปัญหามีสองแนวทางหลัก แนวทางแรกคือการเปลี่ยน address ของแต่ละ VL53 ผ่าน software (เซนเซอร์รองรับการเขียน address ใหม่ลง register 0x8A) แต่วิธีนี้มีจุดอ่อนคือเมื่อชิปถูก reset ทุก address จะถอยกลับไปที่ 0x29 ทำให้ต้องทำขั้นตอน address remapping ใหม่ทุกครั้งที่บูต และต้องใช้ขา XSHUT ในการแยกชิปทีละตัวระหว่างการตั้งค่า ทำให้ขา GPIO ที่ต้องใช้รวมเท่ากับการใช้ multiplexer อยู่ดี

แนวทางที่สองที่โครงงานนี้เลือกใช้คือ I²C multiplexer ตัวรุ่น TCA9548A ของบริษัท Texas Instruments ซึ่งเป็น 8-channel switch ที่ทำหน้าที่เป็น "ตัวสับสายไฟ" ระหว่าง upstream bus (ที่ master ESP32-P4 ใช้) และ downstream channel ทั้งแปดช่อง การเลือก channel ทำได้โดยส่ง control byte (1 byte) ไปยัง address ของ TCA9548A เอง (0x70) bit แต่ละบิตของ control byte แทน channel หนึ่งช่อง เช่น 0x01 = channel 0, 0x02 = channel 1, 0x04 = channel 2, ..., 0x80 = channel 7 และ 0x00 = ปิดทุก channel

**(แทรกรูปภาพ block diagram ของ TCA9548A พร้อม downstream channels)**

**ภาพที่ 2-5** สถาปัตยกรรมการเชื่อมต่อ TCA9548A กับ VL53L0X ทั้ง 6 ตัว

ในโครงงานนี้เซนเซอร์ VL53L0X ตัวที่ 1–6 ถูกต่อเข้ากับ channel 0–5 ของ TCA9548A ตามลำดับ การอ่านค่าระยะของเซนเซอร์ตัวที่ N ทำได้โดย (1) ส่งคำสั่งเลือก channel N ไปที่ TCA9548A (เช่น 0x04 สำหรับ channel 2), (2) ส่งคำสั่ง I²C ไปที่ address 0x29 ตามปกติ ซึ่งจะถูก route ไปที่ VL53L0X ที่อยู่บน channel ที่เลือก, (3) (เลือกได้) ส่ง 0x00 เพื่อปิด channel เพื่อแยกบัส

### 2.4.3 SPI (Serial Peripheral Interface)

SPI เป็นโปรโตคอลแบบ synchronous serial ที่อาศัยสัญญาณ 4 เส้น ได้แก่ MOSI (Master Out, Slave In), MISO (Master In, Slave Out), SCK (Serial Clock) และ CS (Chip Select) แตกต่างจาก I²C ตรงที่ SPI ไม่มี address แต่ใช้สาย CS แยกสำหรับ slave แต่ละตัว ทำให้รองรับความเร็วสูงกว่ามาก และไม่มีปัญหา address conflict โครงงานนี้ใช้ SPI ในการขับจอแสดงผล ST7796S ที่ความเร็ว 40 MHz ใน mode 0 (CPOL=0, CPHA=0) ซึ่งหมายถึง clock idle ที่ระดับ LOW และ data sampling ที่ rising edge อัตราการส่งข้อมูลทฤษฎีคือ 40 Mbps แต่ในทางปฏิบัติประมาณ 32–35 Mbps หลังหักค่า overhead ของ command byte และ DMA setup time

### 2.4.4 MIPI-CSI (Mobile Industry Processor Interface — Camera Serial Interface)

MIPI-CSI เป็นมาตรฐานการส่งข้อมูลจาก image sensor ไปยังโปรเซสเซอร์ที่ใช้กันแพร่หลายในโทรศัพท์มือถือและกล้องดิจิตอล กำหนดโดย MIPI Alliance โครงงานนี้ใช้ MIPI-CSI version 1 (CSI-1) ที่รองรับโดย ESP32-P4 โดยใช้ 2 data lanes บวก 1 clock lane (รวม 3 differential pairs หรือ 6 ขา)

หัวใจของ MIPI-CSI คือเลเยอร์ทางกายภาพ (D-PHY) ที่ทำงานในสองโหมด ได้แก่ **Low-Power (LP) Mode** ที่อัตรา 10 Mbps ใช้ส่งคำสั่งและสถานะ และ **High-Speed (HS) Mode** ที่อัตราหลายร้อย Mbps ต่อ lane ใช้ส่งข้อมูลภาพจริง ความสามารถพิเศษคือสาย differential ที่ใช้แรงดัน common-mode เพียง 200 mV (เทียบกับ LVDS ที่ 1.2 V) ทำให้ใช้พลังงานต่ำมาก

ในโครงงานนี้กล้อง OV5647 ที่ความละเอียด 800×640 พิกเซล อัตรา 50 fps จะสร้างข้อมูล 800 × 640 × 50 × 8 บิต ÷ 2 lanes = 102.4 Mbps ต่อ lane (หรือ 204.8 Mbps รวม) ซึ่งอยู่ในช่วงที่ D-PHY รองรับได้สบาย ๆ เพราะสเปก D-PHY v1.2 รองรับสูงสุด 1.5 Gbps ต่อ lane

**โครงสร้าง packet ของ MIPI-CSI** ประกอบด้วย short packet สำหรับเหตุการณ์ควบคุม (frame start, frame end, line start, line end) และ long packet สำหรับ payload ข้อมูลพิกเซลจริง แต่ละ long packet มี header 4 byte (containing data type, word count, virtual channel) ตามด้วย payload และ checksum 2 byte ผู้รับสัญญาณ (CSI host controller บน ESP32-P4) จะตรวจสอบ checksum และ route packet ไปยัง ISP สำหรับการประมวลผลภาพต่อไป

### 2.4.5 UART (Universal Asynchronous Receiver/Transmitter)

UART เป็นโปรโตคอล serial communication แบบ asynchronous ที่ไม่มีสาย clock แยก แต่อาศัย baud rate ที่ทั้งสองฝ่ายตกลงไว้ล่วงหน้า โครงสร้างของ frame UART มาตรฐานที่โครงงานนี้ใช้ (8-N-1) ประกอบด้วย start bit (logic 0) 1 bit, data 8 bit (LSB first), parity bit (ไม่ใช้ในกรณีนี้ = N) และ stop bit (logic 1) 1 bit รวม 10 bit ต่อ byte ที่ baud rate 9600 bps จึงสามารถส่งข้อมูลได้ 960 bytes/sec

โมดูลขยายเสียง DY-HV20T ที่ใช้ในโครงงานสื่อสารผ่าน UART ที่ baud rate 9600 bps ในรูปแบบ command frame ที่บริษัทผู้ผลิตกำหนด มีโครงสร้างดังนี้: header byte (0xAA), command byte (เช่น 0x07 = play track), length byte, data bytes และ checksum byte ทุก frame ที่ส่งจาก ESP32-P4 ไปยัง DY-HV20T ตามรูปแบบนี้จะกระตุ้นการกระทำที่กำหนด เช่น เล่นเพลง track หมายเลขที่ระบุ ปรับระดับเสียง หรือหยุดเล่น

### 2.4.6 SDIO (Secure Digital Input/Output)

SDIO เป็นโปรโตคอลที่ขยายมาจาก SD card บัสเพื่อรองรับ peripheral อื่นนอกเหนือจาก storage โครงงานนี้ใช้ SDIO ในการเชื่อมต่อระหว่าง ESP32-P4 (host) และ ESP32-C6 (WiFi co-processor) ผ่านสถาปัตยกรรมที่บริษัท Espressif เรียกว่า "ESP-Hosted" บัสประกอบด้วย CMD line, CLK line และ data lines 4 เส้น (D0–D3) ทำงานที่ความเร็ว 40 MHz ใน 4-bit mode รองรับ throughput สูงสุด 160 Mbps ซึ่งเพียงพอต่อการส่งข้อมูล WiFi 2.4 GHz ที่ความเร็วสูงสุด 72 Mbps

ESP-Hosted แบ่ง SDIO ออกเป็นสอง function ได้แก่ Function 0 สำหรับการตั้งค่าและ control register และ Function 1 สำหรับการส่งข้อมูล WiFi packet ระหว่างสอง chip โดย ESP32-P4 ทำหน้าที่เป็นเสมือน application processor ที่ตัดสินใจส่ง/รับ packet ผ่าน RPC ในขณะที่ ESP32-C6 ทำหน้าที่เป็น dedicated WiFi/BLE radio

## 2.5 ทฤษฎีเซนเซอร์

### 2.5.1 เซนเซอร์อินฟราเรดแบบตัดลำแสง (IR Photointerruptor)

เซนเซอร์ตัดลำแสงอินฟราเรดที่ใช้ในโครงงานนี้เป็นแบบ "U-shape Slot Optocoupler" ที่ประกอบด้วย LED อินฟราเรดความยาวคลื่น 940 nm และ phototransistor วางหันหน้ากันในเฟรม U-shape ระยะระหว่างคู่ประมาณ 5 มิลลิเมตร เมื่อมีวัตถุเคลื่อนผ่านระหว่างกลาง ลำแสงจะถูกตัด ทำให้กระแส collector ของ phototransistor ลดลง สัญญาณ output ที่ผ่าน comparator IC LM393 จะเปลี่ยนสถานะจาก HIGH (ปกติ) เป็น LOW (มีวัตถุบัง) อย่างชัดเจน

**ทฤษฎีการดีบาวน์ (Debouncing)** สัญญาณจาก IR sensor มักมี noise ที่เกิดจากการสั่นสะเทือนทางกล (เช่น ตอนเม็ดยาตกผ่านลำแสงและสะท้อนกลับ) หรือจากการรบกวนทางไฟฟ้า (เช่น สัญญาณ PWM ของ servo motor ที่ขับด้วยกระแสสูง) วิธีกำจัด noise นิยมใช้เทคนิค "consecutive sampling" คือการอ่านสัญญาณซ้ำ N ครั้งติดกันที่เปลี่ยนสถานะตรงกันก่อนยอมรับว่ามีการเปลี่ยนสถานะจริง ในโครงงานนี้ใช้ N = 3 หมายความว่าต้องอ่านได้ค่า LOW ติดกัน 3 ครั้งเป็นอย่างน้อยจึงนับว่ามีเม็ดยาผ่าน

ที่อัตราการอ่าน 100 Hz (ทุก 10 มิลลิวินาที) เกณฑ์ N = 3 หมายถึงเวลาดีบาวน์ขั้นต่ำ 30 มิลลิวินาที ซึ่งสั้นพอที่จะจับเหตุการณ์เม็ดยาผ่าน (ปกติใช้เวลา 50–80 มิลลิวินาที) แต่ยาวพอที่จะกรอง noise ที่เกิดเป็น spike สั้น ๆ (ใต้ 10 มิลลิวินาที) ออก

### 2.5.2 เซนเซอร์วัดระยะแบบ Time-of-Flight (VL53L0X)

VL53L0X ของบริษัท STMicroelectronics เป็นเซนเซอร์ระยะแบบ Time-of-Flight (ToF) รุ่น miniature ที่บรรจุแหล่งกำเนิดแสง VCSEL (Vertical-Cavity Surface-Emitting Laser) ความยาวคลื่น 940 nm และ photodiode array แบบ SPAD (Single-Photon Avalanche Diode) ไว้ในแพ็กเกจขนาดเพียง 4.4 × 2.4 × 1.0 มิลลิเมตร

หลักการทำงานเริ่มจากแหล่ง VCSEL ปล่อยพัลส์เลเซอร์อินฟราเรดสั้น ๆ ออกไปยังเป้าหมาย เมื่อแสงสะท้อนกลับมา SPAD array จะตรวจจับและวัดเวลาที่แสงเดินทางทั้งหมด (Δt) ระยะคำนวณจากสมการพื้นฐานทางฟิสิกส์ d = c × Δt / 2 โดย c คือความเร็วแสง 299,792,458 m/s และหารด้วย 2 เพราะแสงเดินทางไปและกลับ ตัวอย่างเช่น เวลาเดินทาง 1 nanosecond แทนระยะทาง 15 เซนติเมตร เซนเซอร์จึงต้องวัดเวลาในระดับ picosecond เพื่อให้ได้ความละเอียดในระดับมิลลิเมตร

**โหมดการทำงานของ VL53L0X** มีทั้งสิ้น 4 โหมด ได้แก่ Default Mode (1.2 m, ±3% accuracy), Long Range Mode (2.0 m), High Speed Mode (33 ms response, 1.2 m) และ High Accuracy Mode (200 ms, ±1% accuracy) โครงงานนี้ใช้ Default Mode ที่ความถี่ continuous 50 ms inter-measurement period ซึ่งให้ rate อ่านได้ 20 ครั้ง/วินาที เหมาะสมสำหรับการวัดระดับเม็ดยาในตลับที่ไม่ต้องการ rate สูงมาก

**การแปลงค่าระยะเป็นจำนวนเม็ดยา** กรอบสมการที่ใช้คือ N_pills = N_max - round((d_measured - d_full) / h_pill) + offset โดย N_max คือจำนวนเม็ดยาสูงสุดที่ตลับบรรจุได้ (16 เม็ด), d_full คือระยะที่วัดได้เมื่อตลับเต็ม (~50 มิลลิเมตร), h_pill คือความสูงของเม็ดยาเฉลี่ย (~15 มิลลิเมตร) และ offset คือค่าปรับ fine-tune ในกรณีที่เซนเซอร์มี zero-point ไม่ตรง 100% ค่าเหล่านี้ปรับได้ผ่านหน้า web technician แต่ละโมดูล

**Signal Rate และ Range Status** เซนเซอร์ส่งคืนค่า "signal rate" ที่บอกความแรงของสัญญาณสะท้อนเป็นหน่วย MCPS (Million Counts Per Second) ค่าต่ำเกินไป (ต่ำกว่า 0.25 MCPS) บ่งบอกว่าวัตถุไกลเกินไปหรือไม่สะท้อนแสง ค่า "range status" บอกประเภทของ error เช่น 0 = valid, 1 = sigma fail, 2 = signal fail, 4 = phase fail โครงงานนี้ตรวจสอบ status เป็น 0 และ raw value > 0 เท่านั้นจึงยอมรับการอ่าน

### 2.5.3 หน้าจอสัมผัสแบบ Capacitive (FT6336U)

ชิป FT6336U ของบริษัท FocalTech เป็น capacitive touch controller รองรับการสัมผัส 2 จุดพร้อมกัน (2-finger multi-touch) ความละเอียด 12 บิต (4096 × 4096) อ่านพิกัดผ่าน I²C ที่ address 0x38

หลักการทำงานของ capacitive touch อาศัยการเปลี่ยนแปลงของ capacitance เมื่อนิ้วเข้าใกล้เซนเซอร์ มีสองรูปแบบหลัก ได้แก่ **Mutual Capacitance** ที่วัด capacitance ระหว่าง electrode ขับ (drive lines, แกน X) และ electrode รับ (sense lines, แกน Y) เมื่อนิ้วแตะใกล้จุดตัดของแกน capacitance ระหว่างสองแกนจะลดลง (เพราะนิ้วดูดสนามไฟฟ้า) ทำให้สามารถระบุพิกัดได้แม่นยำ และ **Self Capacitance** ที่วัด capacitance ของแต่ละ electrode กับ ground แยกกัน ใช้สำหรับโหมด proximity (ตรวจจับนิ้วก่อนแตะจริง)

ข้อมูลการสัมผัสบน FT6336U อ่านได้จาก register block เริ่มที่ 0x02 ขนาด 6 byte ประกอบด้วย จำนวนการสัมผัส (touch count, byte 0), พิกัด X high byte (byte 1), พิกัด X low byte (byte 2), พิกัด Y high byte (byte 3), พิกัด Y low byte (byte 4) และ touch ID + event flag (byte 5) ค่าพิกัด X และ Y ที่อ่านได้เป็นค่าดิบ ต้องผ่านการ map กับขนาดจอ (480 × 320) เพื่อให้ตรงกับตำแหน่งจริงบนจอ

**ปัญหาที่พบในโครงงาน** เนื่องจาก FT6336U บางตัวที่จำหน่ายในตลาดเป็น batch ที่มี firmware เก่า การอ่าน chip ID register (0xA8) ในขั้นตอน init บางครั้งคืนค่า 0x00 หรือค่าที่ไม่ตรงกับชิปปกติ (0x11, 0x06, 0x64) แต่ register block อื่นยังอ่านได้ปกติ โครงงานนี้แก้ปัญหาโดยทำ "fallback mode" คือถ้า chip ID ไม่ตรงแต่ address ยัง ACK และ register touch count ให้ค่าที่สมเหตุสมผล (0–2 จุด) จะใช้งานต่อได้ พร้อมเพิ่ม debounce ให้เข้มงวดขึ้นเพื่อรองรับ noise ที่อาจมาจาก fallback mode

## 2.6 ทฤษฎี Actuator

### 2.6.1 เซอร์โวมอเตอร์ (Servo Motor SG90)

SG90 เป็นเซอร์โวมอเตอร์ขนาดเล็ก (micro servo) รุ่นเปิดตัวที่ใช้แพร่หลายในงานนักศึกษาและงาน prototype น้ำหนักเพียง 9 กรัม รองรับแรงบิด 1.8 kg·cm ที่แรงดัน 4.8 V ซึ่งเพียงพอต่อการขับฝาเปิดของตลับยาที่มีแรงต้านต่ำ ภายในประกอบด้วยมอเตอร์ DC ขนาดเล็ก, เกียร์ทดรอบ (โดยปกติเป็นเกียร์พลาสติกหรือ metal), pot meter ที่อ่านมุมการหมุนเป็น feedback และ control circuit ที่เปรียบเทียบมุมปัจจุบันกับมุม target

**การควบคุมเซอร์โวด้วย PWM** SG90 ทำงานด้วยสัญญาณ PWM ที่ความถี่ 50 Hz (period 20 มิลลิวินาที) ความกว้างของพัลส์ HIGH ในแต่ละรอบบ่งบอกตำแหน่งมุม โดยทั่วไปคือ 0.5 มิลลิวินาที = 0°, 1.5 มิลลิวินาที = 90°, 2.5 มิลลิวินาที = 180° สมการ pulse_us = 500 + (angle × 2000 / 180) แปลงมุมที่ต้องการเป็นความกว้างพัลส์ จากนั้นต้องคำนวณค่า "off count" สำหรับ register ของ PCA9685 ที่มีความละเอียด 12 บิต (4096 ระดับ) ตามสมการ off_count = pulse_us × 4096 / 20000 = pulse_us × 0.2048

**ข้อจำกัดของ SG90** จุดอ่อนหลักคือกระแส idle ขณะอยู่นิ่งสูงประมาณ 100–200 mA (เพราะ control circuit คอย hold ตำแหน่งตลอด) และกระแสเริ่มต้นการหมุน (stall current) สูงถึง 700 mA หากใช้ servo 6 ตัวพร้อมกันในจังหวะเริ่มจ่ายยาแบบ scheduled dose จะมี surge current สูงสุดประมาณ 4 A ซึ่งเป็นสาเหตุที่โครงงานนี้ใช้ power supply 5V ขนาด 10 A เพื่อรับมือกับกรณีนี้

### 2.6.2 PCA9685 (16-channel PWM Driver)

PCA9685 ของบริษัท NXP เป็น I²C-controlled PWM driver ที่รองรับ 16 channel แต่ละ channel มีความละเอียด 12 บิต (4096 ระดับ) และความถี่สามารถปรับได้ในช่วง 24 Hz – 1526 Hz การที่ PCA9685 มี oscillator ในตัว 25 MHz ทำให้สามารถ generate สัญญาณ PWM ได้แม่นยำโดยไม่ต้องอาศัย CPU ของ ESP32-P4 ในการประมวลผลเลย

การคำนวณค่า prescaler ที่กำหนดความถี่ใช้สมการ prescale = round(25000000 / (4096 × f_pwm)) - 1 ที่ f_pwm = 50 Hz (ตามสเปกของ SG90) ค่า prescale = round(25000000 / 204800) - 1 = round(122.07) - 1 = 121 ซึ่งโครงงานนี้เขียนค่า 121 ลงไปใน register PRESCALE (0xFE) ของ PCA9685

**ขั้นตอนการ initialize** ตามที่กำหนดในข้อมูลทางเทคนิคของ PCA9685 ประกอบด้วย (1) เขียน MODE1 = 0x10 เพื่อเข้าสู่ sleep mode (จำเป็นเพราะ PRESCALE จะเขียนได้เฉพาะตอน sleep), (2) เขียน PRESCALE = 121, (3) เขียน MODE1 = 0x00 เพื่อ wake-up, (4) รอ ≥500 µs ให้ oscillator stabilize, (5) เขียน MODE1 = 0xA0 เพื่อเปิด auto-increment register address (ทำให้ผ่านบรรทัด ON_L, ON_H, OFF_L, OFF_H ของแต่ละ channel เป็นชุดได้)

โครงงานนี้ใช้ PCA9685 ที่ I²C address 0x40 (default) สำหรับขับเซอร์โว 6 ตัวบน channel 0–5 โดย channel 6–15 สำรองไว้สำหรับการขยายในอนาคต

## 2.7 ระบบ Computer Vision

### 2.7.1 OV5647 image sensor

OV5647 เป็นเซนเซอร์ภาพ CMOS รุ่นเก่าแต่ยังนิยมใช้ของบริษัท OmniVision ความละเอียดสูงสุด 5 ล้านพิกเซล (2592 × 1944 พิกเซล หรือเรียกว่า 5 MP) เซลล์รับแสง (pixel) ขนาด 1.4 × 1.4 µm ที่จัดเรียงในรูปแบบ Bayer pattern แบบ BGGR (Blue-Green-Green-Red) ในแต่ละ 2×2 quad ของพิกเซล 4 ตัว มี 2 พิกเซลเขียว 1 พิกเซลแดง และ 1 พิกเซลน้ำเงิน

การควบคุมและการตั้งค่าของกล้องทำผ่านโปรโตคอล SCCB (Serial Camera Control Bus) ของ OmniVision ซึ่ง compatible กับ I²C address 7-bit ของ OV5647 คือ 0x36 register address ขนาด 16 บิต ทำให้รองรับ register สูงสุด 65536 ตัว (จริงๆ ใช้แค่ประมาณ 200 ตัว) Product ID ของ OV5647 อ่านได้จาก register pair 0x300A (high byte = 0x56) และ 0x300B (low byte = 0x47) รวมเป็น 0x5647 ซึ่งคือชื่อรุ่น

ในโครงงานนี้กล้องถูกตั้งค่าให้ทำงานที่ความละเอียด 800 × 640 พิกเซล อัตรา 50 fps ในรูปแบบ RAW8 (Bayer) สาเหตุที่ไม่ใช้ความละเอียดเต็ม 5 MP คือ (1) โครงงานต้องการเฟรมเรตสูงเพื่อให้ MJPEG stream ลื่นไหล ไม่ใช่ความละเอียดสูง, (2) memory ของ ESP32-P4 (32 MB PSRAM) จำกัด ภาพ raw 5 MP จะใช้ buffer 5 MB ต่อเฟรม ซึ่งทำให้ frame buffer 3 ตัว ใช้ 15 MB เกือบครึ่งของ PSRAM, (3) bandwidth ของ MIPI-CSI 2 lane ที่ ESP32-P4 ออกแบบรองรับเหมาะกับ 1-2 MP มากกว่า 5 MP

### 2.7.2 ISP Pipeline บน ESP32-P4

ISP (Image Signal Processor) ที่ฝังในตัว ESP32-P4 เป็นบล็อกฮาร์ดแวร์ที่รับข้อมูล Bayer raw จาก MIPI-CSI ผ่าน CSI bridge แล้วประมวลผลตาม pipeline ดังนี้

**ขั้นที่ 1 — Demosaic** เป็นการแปลง Bayer pattern (พิกเซลละ 1 สี) เป็น RGB888 (พิกเซลละ 3 สี) โดยอาศัยการ interpolate ค่าสีที่หายจากเซลล์ข้างเคียง อัลกอริทึมที่ใช้ใน ESP32-P4 เป็นแบบ bilinear interpolation ซึ่งคำนวณค่า G ที่ตำแหน่ง R โดยเฉลี่ย G สี่เซลล์รอบ ๆ และคำนวณค่า B โดยเฉลี่ย B ของเซลล์ทแยงมุม ผลลัพธ์เป็นภาพ RGB888 ที่มีความละเอียด full

**ขั้นที่ 2 — CCM (Color Correction Matrix)** เป็นการคูณเวกเตอร์สี [R, G, B] กับ matrix 3×3 เพื่อปรับ color gamut และ saturation สมการคือ [R'; G'; B'] = M × [R; G; B] ที่ M เป็น matrix ที่ผู้พัฒนากำหนด โครงงานนี้ใช้ matrix ที่บูสต์ saturation (แถวทแยง 2.4 และ off-diagonal -0.5) เพื่อชดเชยภาพที่ดู washout จาก demosaic

**ขั้นที่ 3 — RGB to YUV422 Conversion** สำหรับ output YUV422 (ที่ JPEG hardware encoder ใน ESP32-P4 ใช้เป็น input format) ต้องแปลง RGB888 ผ่าน standard ITU-R BT.601 matrix Y = 0.299R + 0.587G + 0.114B, U = -0.169R - 0.331G + 0.500B + 128, V = 0.500R - 0.419G - 0.081B + 128 พร้อม downsample U และ V ในแกน horizontal เหลือครึ่งของ rate

ผลลัพธ์ของ pipeline ทั้งสามขั้นคือภาพ YUV422 ขนาด 800 × 640 ที่ส่งต่อให้ JPEG encoder ผ่าน 2D-DMA โดยไม่ผ่าน CPU เลย ทำให้ throughput ของระบบสูงมาก

### 2.7.3 JPEG Hardware Encoder

ESP32-P4 มี JPEG codec ในตัวที่รองรับทั้ง encode (image → JPEG file) และ decode (JPEG → image) บล็อกนี้ทำงานตามมาตรฐาน JPEG baseline ของ ITU-T T.81 ขั้นตอนหลักของการ encode ประกอบด้วย

**ขั้นที่ 1 — DCT (Discrete Cosine Transform)** ภาพถูกแบ่งเป็น block ขนาด 8×8 พิกเซล (สำหรับ Y) และ 8×8 (สำหรับ U, V ที่ subsample แล้ว) แต่ละ block ผ่าน DCT แปลงข้อมูลจาก spatial domain (พิกเซล) เป็น frequency domain (ค่าสัมประสิทธิ์) coefficient ที่อยู่มุมบนซ้ายแทน DC component (ค่าเฉลี่ย) ส่วนที่เหลือแทน AC component ที่ความถี่สูงขึ้นเรื่อย ๆ

**ขั้นที่ 2 — Quantization** หาร coefficient ที่ได้ด้วยตาราง quantization table 64 ค่า ค่ามากในตารางสำหรับความถี่สูง (ที่สายตามนุษย์มองไม่เห็นชัด) ทำให้ coefficient ที่ความถี่สูงถูก zero out หลังหารและปัดเศษ ขั้นตอนนี้คือที่มาของ "lossy compression"

**ขั้นที่ 3 — Run-Length + Huffman Coding** เรียง coefficient จาก zigzag pattern (เริ่มจาก DC แล้วไหลขึ้นลง) ทำให้ coefficient ที่เป็น 0 ถูกจัดกลุ่มกันต่อท้าย encode ด้วย Huffman code ที่ให้ความยาว bit สั้นกับค่าที่พบบ่อย และยาวกับค่าที่พบน้อย

ผลลัพธ์ของขั้นทั้งสามคือ stream ของ bit ที่บรรจุอยู่ในไฟล์ .jpg ตามมาตรฐาน JFIF JPEG hardware encoder ของ ESP32-P4 ทำได้ที่ throughput ประมาณ 50 MB/s (ภาพ raw input) ที่ quality factor ปานกลาง ภาพ 800×640 ใช้เวลา encode ประมาณ 10–15 ms ต่อเฟรม

### 2.7.4 Software 90° Rotation

เนื่องจากกล้องในโครงงานติดตั้งในแนวนอน (landscape) แต่ภาพที่ส่งให้ผู้ใช้ผ่าน Telegram ต้องการเป็นแนวตั้ง (portrait) จึงต้องหมุนภาพ 90° ก่อน encode แม้ว่า Telegram จะอ่านค่า EXIF orientation tag ได้ แต่ Telegram จะลบ tag นี้ออกตอน upload ทำให้ผู้รับเห็นภาพในแนวที่ผิด การ rotate pixel โดยซอฟต์แวร์จึงเป็นวิธีเดียวที่ใช้ได้

อัลกอริทึม 90° CCW (Counter-Clockwise) ที่โครงงานนี้ใช้คือ dst[dy][dx] = src[src_h - 1 - dx][dy] ซึ่งหมายถึงว่า แต่ละพิกเซลปลายทางจะถูกอ่านจากตำแหน่งที่หมุน 90° ในภาพต้นทาง implementation จริงต้อง handle YUV422 packed format (Y0 U Y1 V) ที่ U/V ถูก share ระหว่างพิกเซลคู่ ทำให้การ rotate ต้องระวัง chroma sampling

ใช้เวลา ~25 ms ต่อเฟรม 800×640 บน ESP32-P4 ที่ 360 MHz ซึ่งต่ำพอที่จะรักษาเฟรมเรต 50 fps ไม่ให้ทำงานช้าลง memory ที่ใช้สำหรับ rotation buffer คือ 800 × 640 × 2 = 1,024,000 bytes ใน PSRAM

## 2.10 สรุปบทที่ 2

บทที่ 2 ได้กล่าวถึงทฤษฎีและเทคโนโลยีสำคัญที่นำมาใช้ในโครงงาน "เครื่องจ่ายยาอัตโนมัติ" ครอบคลุมตั้งแต่สถาปัตยกรรมของไมโครคอนโทรลเลอร์ ESP32-P4 ที่ใช้เป็นแกนกลาง รวมถึงเฟรมเวิร์ก ESP-IDF และระบบปฏิบัติการ FreeRTOS ที่จัดการการทำงานหลายภาระงานอย่างเป็นระบบ ตามด้วยโปรโตคอลการสื่อสารฮาร์ดแวร์ครบ 6 รูปแบบ ได้แก่ I²C, SPI, MIPI-CSI, UART, SDIO และ LEDC จากนั้นได้อธิบายหลักการทำงานของเซนเซอร์ทั้งสามชนิด คือ IR Photointerruptor สำหรับนับเม็ดยา, VL53L0X สำหรับวัดระดับยาคงเหลือ และ FT6336U สำหรับหน้าจอสัมผัส ส่วนของ Actuator ครอบคลุมเซอร์โวมอเตอร์ SG90 และ PCA9685 PWM driver

ต่อมาได้กล่าวถึงระบบ computer vision ของ ESP32-P4 ที่ครอบคลุม OV5647 image sensor, ISP pipeline, JPEG hardware encoder และอัลกอริทึมหมุนภาพ 90° ซอฟต์แวร์ ปิดท้ายด้วยเทคโนโลยี IoT และคลาวด์ทั้ง 5 บริการ ได้แก่ MQTT, NETPIE 2020, HTTPS/TLS, Telegram Bot API และ Google Apps Script Webhook พร้อมทบทวนงานวิจัย 9 ฉบับที่มีลักษณะใกล้เคียง และเปรียบเทียบจุดแตกต่างของโครงงานนี้กับงานเหล่านั้น

ในบทที่ 3 จะกล่าวถึงวิธีการดำเนินงานเชิงปฏิบัติ ตั้งแต่การวางแผน ออกแบบฮาร์ดแวร์ ออกแบบซอฟต์แวร์ และทดสอบ พร้อมแสดง Bill of Materials, schematic, flowchart และ sequence diagram ของระบบทั้งหมด
