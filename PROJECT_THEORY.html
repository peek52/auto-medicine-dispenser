บทที่ 1

บทนำและวัตถุประสงค์

การลืมรับประทานยาเป็นปัญหาที่พบบ่อยในผู้สูงอายุและผู้ป่วยโรคเรื้อรัง ส่งผลต่อประสิทธิภาพการรักษาและคุณภาพชีวิต โครงการนี้พัฒนาเครื่องจ่ายยาอัตโนมัติที่ผสมผสานระหว่าง embedded system, IoT, และ cloud computing เพื่อช่วยให้ผู้ใช้รับประทานยาตรงเวลาและลดภาระของผู้ดูแล

ปัญหาที่ต้องการแก้

  • ลืมรับประทานยา — ไม่มีการแจ้งเตือนหรือนับจำนวนเม็ดยา
  • รับประทานผิดมื้อ/ผิดขนาด — สาเหตุของยา overdose หรือขาด
  • ผู้ดูแลตรวจสอบยาก — ต้องเดินทางมาดูถึงที่
  • ไม่มีบันทึกประวัติ — สำหรับแพทย์ตรวจสอบ adherence

วัตถุประสงค์

  1. ออกแบบและสร้างเครื่องจ่ายยาที่จัดเก็บยาได้ 6 ชนิด และจ่ายยาอัตโนมัติตามตารางเวลา
  2. นับจำนวนเม็ดยาที่จ่ายได้แม่นยำด้วย IR sensor ป้องกัน over/under dispense
  3. วัดปริมาณยาคงเหลือในแต่ละช่อง (stock) ด้วย ToF sensor และ sync ขึ้น cloud
  4. เชื่อมต่อกับ NETPIE 2020, Telegram Bot, และ Google Sheets เพื่อให้ผู้ดูแลตรวจสอบจากระยะไกล
  5. มีกล้องสำหรับดูสถานะเครื่องจากระยะไกล (live + capture on demand)
  6. มี UI บนจอ touchscreen สำหรับใช้งาน standalone โดยไม่ต้องพึ่ง smartphone

บทที่ 3

ทฤษฎี ESP32-P4 (Microcontroller)

ESP32-P4 เป็นชิป microcontroller สถาปัตยกรรม RISC-V ออกแบบโดย Espressif Systems รองรับงานประมวลผลภาพ, จอ HMI, และเชื่อมต่อ peripheral หลากหลาย เหมาะกับโครงการที่ต้องการ AI / vision / multimedia

FreeRTOS — Real-Time Operating System

โปรเจกต์ใช้ FreeRTOS สำหรับ task scheduling แบบ preemptive priority-based. แต่ละ task ทำงานอิสระ (ดูแลโดย scheduler) และสื่อสารกันผ่าน queue, semaphore, mutex.

หลักการสำคัญ:

  • Priority — task ที่มี priority สูงกว่าจะ preempt task ที่ priority ต่ำกว่า
  • Round-robin — task ที่ priority เท่ากันจะแบ่ง CPU แบบเวียน
  • Blocking — task ที่รอ event (queue, semaphore) จะถูก suspend ไม่กิน CPU
  • Mutex — ป้องกันสองตัวเข้าถึง resource เดียวกันพร้อมกัน (เช่น I²C bus)
บทที่ 4

โปรโตคอล I²C และการขยายบัส

I²C (Inter-Integrated Circuit) เป็นโปรโตคอลแบบ 2-wire (SDA + SCL) ใช้แบบ master-slave ที่นิยมมากในงาน embedded เพราะเดินสายน้อยและรองรับ device หลายตัวบนบัสเดียว

หลักการทำงาน

  • SDA (Serial Data) — ส่งข้อมูล bit-by-bit แบบ open-drain (ต้องมี pull-up)
  • SCL (Serial Clock) — clock ที่ master เป็นผู้กำเนิด
  • Address 7-bit — แต่ละ slave มี address ไม่ซ้ำ master ส่ง address + bit R/W เพื่อระบุ slave
  • ความเร็วมาตรฐาน 100 kHz (Standard mode), 400 kHz (Fast mode), 1 MHz (Fast mode plus)
SDA ─┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──┐ ┌──┐ │ │ A6 │ │ A5 │ │ .. │ │A0│ │RW│ ACK ... SCL ──┘ └──────┘ └──────┘ └──────┘ └──┘ └──┘ ─── ↑ ↑ START SLAVE ACK

ปัญหา: Address ซ้ำกัน → ใช้ I²C Multiplexer

VL53L0X ทุกตัวมี default address เดียวกัน (0x29) ทำให้ต่อบัสเดียวกันไม่ได้. แก้ปัญหาด้วย TCA9548A ซึ่งเป็น I²C mux 8 ช่อง — เลือกได้ทีละช่องผ่าน register ของ mux

วิธีใช้ TCA9548A

  1. Master เขียน byte ไปที่ address ของ mux (0x70) เพื่อเลือก channel (bit 0..7)
  2. หลังจากเลือก channel, การสื่อสารต่อไปจะ "ทะลุ" ไปยัง slave ในช่องนั้น
  3. เปลี่ยน channel ใหม่เมื่อจะคุยกับ slave อื่น

ในโปรเจกต์เราใช้ TCA9548A เพื่อต่อ VL53L0X 6 ตัว และในขณะ boot ใช้ XSHUT (sleep pin) ของแต่ละตัวเพื่อ reassign address เป็น 0x71-0x76 ทำให้สามารถสลับช่องได้รวดเร็วโดยไม่ต้องเปลี่ยน address ทุกครั้ง

ข้อควรระวัง: Pull-up resistor ของ I²C ต้องเหมาะสมกับ bus capacitance ปกติใช้ 4.7 kΩ ที่ 100 kHz, 2.2 kΩ ที่ 400 kHz หากค่ามากเกินไป rise time จะช้าทำให้ noise margin แคบและเกิด error ได้ง่าย

บทที่ 6

MIPI-CSI และการประมวลผลภาพ

MIPI-CSI (Camera Serial Interface) ของ MIPI Alliance เป็นมาตรฐาน high-speed differential serial สำหรับเชื่อมต่อ image sensor กับ application processor นิยมในโทรศัพท์มือถือและ embedded camera

คุณสมบัติ

  • Differential signaling — แต่ละ lane มี 2 สาย (D±) ลด noise
  • Multi-lane — โปรเจกต์เราใช้ 2-lane ได้ throughput ~ 1 Gbps/lane
  • D-PHY layer — physical layer พร้อม high-speed mode (HS) และ low-power mode (LP)
  • Configuration ผ่าน SCCB — Serial Camera Control Bus (เป็น I²C ที่ปรับเล็กน้อย)

กล้อง OV5647

เซ็นเซอร์ภาพ 5 MP ของ OmniVision รุ่นเดียวกับที่ใช้ใน Raspberry Pi Camera v1 รองรับ resolution สูงสุด 2592×1944 ที่ 15 fps. ในโปรเจกต์เราใช้ 800×640 RAW8 ที่ 50 fps เพื่อความ responsive ของ live stream

JPEG encoding (hardware-accelerated)

ESP32-P4 มี JPEG codec ในตัว ทำให้สามารถแปลง raw frame เป็น JPEG ได้โดยไม่กิน CPU. โปรเจกต์เราใช้สำหรับ:

  • /capture — ส่งภาพ JPEG ไป Telegram bot
  • /stream — MJPEG streaming ผ่าน HTTP (Multipart MIME)
บทที่ 7

UART (Universal Asynchronous Receiver/Transmitter)

UART เป็นโปรโตคอล serial แบบ asynchronous (ไม่มี clock) ใช้ start/stop bit ในการ frame ข้อมูล โปรเจกต์เราใช้สำหรับสื่อสารกับโมดูลเสียง DY-HV20T

หลักการ

  • 2 สาย: TX (transmit) + RX (receive)
  • Frame: 1 start bit + 8 data bits + (parity) + stop bit
  • Baud rate ต้องตรงกันทั้งสองฝั่ง (ใช้ 9600 หรือ 115200 ทั่วไป)
  • โปรเจกต์ใช้ UART1 ที่ baud 9600 ส่ง command 5-byte ให้ DY-HV20T เล่น track

ตัวอย่าง command: AA 07 02 00 0F BD แปลว่า "เล่น track 15"

บทที่ 8

ทฤษฎี Sensor: IR และ ToF

1️⃣ IR Photointerruptor (นับเม็ดยา)

เซ็นเซอร์ IR ทำงานบนหลักการ break-beam: LED ส่ง infrared ตรงไปยัง phototransistor ฝั่งตรงข้าม. เมื่อมีวัตถุผ่านลำแสง output จะเปลี่ยนจาก LOW เป็น HIGH (หรือกลับกัน)

การ debounce และ edge detection

เซ็นเซอร์ IR มี noise spike จาก ambient light, EMI, และ signal bouncing ตอนยาผ่าน. โปรเจกต์ใช้สองชั้น:

  1. Debounce: ต้องอ่านได้ค่า LOW ติดต่อกัน ≥ 5 ครั้ง (~2.5 ms) จึงนับว่าเป็นการตรวจจับจริง
  2. Edge detection: ต้องเห็น HIGH ก่อน → LOW (ไม่ใช่ค้าง LOW) เพื่อกันการนับซ้ำเมื่อยาค้างในลำแสง
IF current = LOW AND previous = HIGH AND debounce_count ≥ 5: pill_count += 1 state = "armed"

2️⃣ Time-of-Flight (วัดยอดยาคงเหลือ)

VL53L0X ของ STMicroelectronics วัดระยะทางด้วยหลักการ time-of-flight โดย:

  1. ยิงแสง laser 940 nm (Class 1, ปลอดภัยต่อตา)
  2. วัดเวลาที่แสงเดินทางไป-กลับ (round trip)
  3. คำนวณระยะทาง: d = c × Δt / 2 (c = ความเร็วแสง)

VL53L0X ทำงานในระยะ 30-2000 mm มีความแม่นยำ ±3% (ที่ 1 m ความผิดพลาด ≤ 30 mm)

การประยุกต์

ติดตั้ง VL53L0X ที่ฝาด้านบนของช่องเก็บยา วัดระยะลงไปยังกองยา. ยิ่งยาเยอะ → ระยะใกล้ (ค่าน้อย). ยิ่งยาน้อย → ระยะไกล (ค่ามาก)

pill_count_estimate = (max_distance − measured_distance) / pill_height

เนื่องจาก servo ทำให้กระบอกยาสั่น ค่าจึงแกว่ง ±2 mm. โปรเจกต์ใช้ stable-tick filter: ต้องอ่านได้ค่าใกล้เคียงกัน 5 ครั้งต่อกันก่อนจะ sync ขึ้น cloud (ลด MQTT flood)

บทที่ 9

ทฤษฎี Actuator: Servo + PWM

Servo motor ทำงานจาก PWM (Pulse-Width Modulation) — สัญญาณที่มีความถี่คงที่แต่ความกว้างของ pulse แปรผัน. ตำแหน่งของ servo ถูกกำหนดโดย duty cycle

PCA9685 — PWM Driver IC

เนื่องจาก ESP32 มี hardware PWM จำกัด (และต้องใช้ pin ของ MCU ตรง ๆ) เราใช้ PCA9685 ของ NXP ซึ่งเป็น I²C-controlled PWM driver 16 ช่อง:

  • ความละเอียด 12-bit (4096 ระดับต่อ cycle)
  • Frequency ปรับได้ 24-1526 Hz (โปรเจกต์ใช้ 50 Hz สำหรับ servo)
  • ควบคุมผ่าน I²C address 0x40

การคำนวณ duty cycle ของ Servo

Servo มาตรฐานคาดหวัง pulse 1.0-2.0 ms ทุก 20 ms (50 Hz):

  • 1.0 ms = 0° (สุดด้านซ้าย)
  • 1.5 ms = 90° (ตำแหน่งกลาง)
  • 2.0 ms = 180° (สุดด้านขวา)
pulse_us = 1000 + (angle / 180) × 1000 duty_4096 = pulse_us × 4096 × 50 / 1,000,000

ลำดับการจ่ายยา

  1. Servo หมุนจากตำแหน่ง HOME → WORK (ดันกระบอกยาเปิด)
  2. รอ 1.5 วินาที (ให้ยาตกลงราง)
  3. Servo หมุนกลับ HOME (ดันยาผ่าน IR sensor)
  4. IR detection event ระหว่าง 4.5 วินาที (รวมเวลา HOME return + bail margin)
  5. นับยา ทำซ้ำจนครบจำนวน
บทที่ 10

จอ TFT & Capacitive Touch

ST7796S — TFT controller

ST7796S เป็น TFT controller รองรับ resolution 480×320 16-bit color (RGB565) ผ่าน SPI. โปรเจกต์ใช้คำสั่งหลัก:

  • 0x36 MADCTL — กำหนด orientation (landscape/portrait, mirror)
  • 0x2A CASET / 0x2B RASET — กำหนดหน้าต่าง draw
  • 0x2C RAMWR — เขียน pixel data
  • 0x29 DISPON — เปิดจอ

FT6336U — Capacitive Touch

Touch controller capacitive แบบ 2-finger ใช้หลัก mutual capacitance: เมื่อนิ้วแตะจอ ค่า capacitance ของจุดนั้นเปลี่ยน. controller ตรวจจับจุดที่เปลี่ยนแล้ว report (x, y) ผ่าน I²C (address 0x38)

โปรเจกต์ใช้ polling mode (อ่าน register ทุก 20 ms) แทน interrupt — เพื่อความเรียบง่ายและไม่ต้องเดิน INT pin

UI Framework

เขียนเองด้วย C++ ใช้ primitive ระดับ pixel/rectangle (fill_rect, draw_string) บน framebuffer. ปรับ rendering ให้ partial-redraw เพื่อหลีกเลี่ยง flicker:

  • วาดเฉพาะส่วนที่ค่าเปลี่ยน (diff-based)
  • ไม่ clear ทั้งหน้าจอ ทำให้ผู้ใช้ไม่เห็น flash สีดำ
บทที่ 11

โปรโตคอลคลาวด์: MQTT & HTTPS

MQTT — Message Queuing Telemetry Transport

MQTT เป็น lightweight publish-subscribe protocol สำหรับ IoT พัฒนาโดย IBM, มาตรฐาน OASIS

  • Broker — server กลาง (เราใช้ NETPIE 2020)
  • Topic — เส้นทางข้อความแบบ hierarchical (เช่น @shadow/data/update)
  • QoS levels — 0 (at most once), 1 (at least once), 2 (exactly once)
  • Retained messages — Broker เก็บข้อความล่าสุดของ topic

บทที่ 12

อัลกอริทึมสำคัญ

1. Edge-Triggered IR Detection with Debouncing

const int DEBOUNCE_SAMPLES = 5; int consecutive_low = 0; bool ir_seen_high = false; while (in_dispense_window) { bool low = read_ir() == 0; if (!low) { ir_seen_high = true; consecutive_low = 0; } else if (ir_seen_high) { if (++consecutive_low >= DEBOUNCE_SAMPLES) { pill_count++; ir_seen_high = false; consecutive_low = 0; } } }

อัลกอริทึมนี้แก้ 2 ปัญหา: (1) noise spike → ใช้ debounce, (2) ยาค้างในลำแสง → ต้องเห็น HIGH ก่อนนับครั้งถัดไป

2. ToF Stable-Tick Filter (ลด MQTT flood)

const int STABLE_TICKS = 5; int last_published = -1; int candidate = -1; int stable_count = 0; every 1 second: new_count = read_pill_count_from_vl53(); if (new_count == candidate) stable_count++; else { candidate = new_count; stable_count = 1; } if (stable_count >= STABLE_TICKS && candidate != last_published) { publish_to_netpie(candidate); last_published = candidate; }

กันการ publish flood เมื่อค่า oscillate (เช่น 14↔16 จาก servo vibration)

3. Mid-Cycle Bail (หยุดถ้ายาหมด)

เมื่อ servo ทำงานครบ cycle แต่ไม่เจอยาผ่าน IR ภายใน 4.5 วินาที → ยุติ loop ทันที (ไม่ต้องวนต่อจนครบจำนวน)

4. I²C Race Mitigation (retry wrapper)

ESP-IDF v5.3.2 มี race condition ใน i2c_master ISR ที่ทำให้ INVALID_STATE หรือ TIMEOUT. แก้ที่ root level ด้วย retry:

esp_err_t i2c_safe_op(...) { esp_err_t r; for (int retry = 0; retry < 3; retry++) { r = i2c_master_transmit(...); if (r == ESP_OK) return r; if (r != ESP_ERR_INVALID_STATE && r != ESP_ERR_TIMEOUT) return r; esp_rom_delay_us(2000); // 2 ms backoff } return r; }
บทที่ 13

Fault-tolerance & Watchdog Theory

ระบบ embedded ที่ทำงานต่อเนื่อง 24/7 ต้องมีกลไกฟื้นตัวจากความผิดพลาด เราใช้ Three-Tier Watchdog Strategy

Tier 1: Task Watchdog Timer (TWDT)

FreeRTOS task ต้อง "feed" watchdog ภายในเวลาที่กำหนด (30 s). ถ้าไม่ feed → panic → reboot

Tier 2: Liveness Canary Task

Task ที่ priority ต่ำที่สุดทำงานเพียงอย่างเดียวคือ feed TWDT ทุก 2 s. ถ้า scheduler ปกติ task นี้ ต้อง ได้ CPU time

Tier 3: esp_timer Zombie Detector

ใช้ hardware timer (esp_timer task) ตรวจสอบ heartbeat counter ทุก 10 s. ถ้า counter ไม่ tick > 120 s → call esp_restart() โดยตรง

┌─ Tier 3: esp_timer (HW timer) ─ 120 s zombie threshold │ └─ Tier 2: Liveness canary task ─ 2 s heartbeat │ └─ Tier 1: TWDT ─ 30 s panic threshold └─ ระดับล่างฟื้นไม่ได้ → ระดับบนยังจับได้
