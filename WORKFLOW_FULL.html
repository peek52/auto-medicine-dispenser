บทนำ + ที่มา

โปรเจกต์นี้คือ เครื่องจ่ายยาอัตโนมัติ (Automated Pill Dispenser) ที่ใช้ ESP32-P4-Nano เป็นสมองหลัก รวมระบบ: กล้อง MIPI-CSI (OV5647) สำหรับถ่ายภาพยืนยันการจ่าย, จอ TFT 480×320 ที่มี touch, servo 6 ช่องสำหรับหมุน carousel จ่ายยา, IR sensor 6 ตัวสำหรับตรวจการตกของยา, RTC สำหรับนาฬิกาแม่นยำ, WiFi + NETPIE MQTT + Telegram bot + Google Sheets สำหรับการแจ้งเตือนและ log ระยะไกล

เป้าหมายของระบบ:
  • จ่ายยาตามเวลาที่ผู้ใช้ตั้งไว้ (7 slot ต่อวัน — เช้าก่อน/หลังอาหาร, เที่ยงก่อน/หลัง, เย็นก่อน/หลัง, ก่อนนอน)
  • ตรวจสอบว่ายาถูกหยิบไปจริงผ่าน IR beam
  • แจ้งเตือน Telegram พร้อมรูปถ่ายเมื่อถึงเวลาจ่าย
  • log ทุก event ไป Google Sheets เก็บประวัติ
  • ทำงานต่อได้เมื่อ WiFi หลุด (queue events ใน NVS)
  • UI บนหน้าจอ touch สำหรับ on-device control

Shared resources + sync

  • I2C bus mutex (g_i2c_mutex): กั้นทุก I2C op — ทุก task ที่ใช้ I2C ต้อง take ก่อน
  • Dispense state mutex (s_dispense_state_mux): กั้น s_waiting_confirm, s_dispense_approved, s_pending_slot_idx
  • Display mutex (s_disp_mutex): กั้น fill_rect static buffer cache
  • IP mux (s_ip_mux): กั้น s_ip string write
  • NETPIE shadow mutex: กั้น parse_shadow + setter; netpie_get_shadow() คืน double-buffered snapshot
  • Offline queue mutex (s_lock): กั้น queue arrays + NVS save/load

ทำไม camera ถูก init แบบ "lazy" / "early"?
Camera SCCB ใช้ I2C bus เดียวกับ peripherals อื่น ถ้าเรา init camera ตอนที่ VL53 หรือ Touch กำลังโผล่ขึ้นมา จะชนกันแล้ว bus wedge. โค้ดเลย init camera ตั้งแต่ตอน I2C "บริสุทธิ์" สดๆ ก่อน peripheral ตัวอื่น แล้วค่อย restore VL53 XSHUT หลัง SCCB เสร็จ

การจ่ายยา (Dispense flow)

Scheduled dispense (อัตโนมัติตามเวลา)

dispenser_task (period ~10s): ├─ get current HH:MM from system time ├─ for each slot 0..6: │ ├─ parse slot_time (e.g., "08:00") │ ├─ delta = cur_minute - slot_minute (with day wrap) │ ├─ skip if delta < 0 or > 5 min ← grace window │ ├─ skip if s_slot_last_fire[slot] within last 12h ← refire guard │ ├─ check meds assigned to slot (sh->med[i].slots bitmask) │ ├─ check stock count > 0 │ ├─ send Telegram + audio "เตรียมยา" │ ├─ set s_waiting_confirm = true, s_pending_slot_idx = slot │ └─ wait for user CONFIRM via UI / Telegram │ ├─ if confirmed → execute_dispense(slot): │ ├─ pause vl53 polling (lock I2C) │ ├─ for each med in slot: │ │ ├─ pcf8574_block_during_servo_ramp(true) ← suppress phantom │ │ ├─ pca9685_go_work(med) ← ramp 1.35s │ │ ├─ unblock + IR poll 7.5s for pill drop │ │ ├─ debounce 3 LOW samples = real pill │ │ ├─ pca9685_go_home(med) ← ramp 1.35s │ │ └─ count++ if detected │ ├─ play audio "จ่ายเสร็จ" │ ├─ send Telegram photo + text summary │ └─ log to Google Sheets (or queue if offline)

Manual dispense (กดผ่าน UI / Telegram /dispense)

เกือบเหมือน scheduled แต่:

  • ข้าม "waiting confirm" step
  • รัน manual_dispense_task เป็น task แยก ไม่ block scheduler
  • รองรับ qty arg: qty=1..16 = จ่ายจำนวนเม็ดนี้, qty=100 = eject all
  • Servo-trust mode: ถ้า IR ของ slot นั้น disabled → trust ทุก servo cycle = 1 pill

Return-pill flow (คืนยา)

หลักการเดียวกับ manual dispense — servo ทำงานสลับทิศ (work → home) → IR ตรวจการตกของยา → นับเข้า stock

การตรวจจับยา (IR + PCF8574)

Hardware path

IR module (FC-51, 6 ตัว, 1 ต่อ slot) │ VCC 3.3V/5V │ GND │ OUT (open-collector + internal pull-up ~10kΩ) │ ▼ PCF8574 (0x20) — quasi-bidirectional, internal pull-up ~100µA │ ▼ I2C │ ESP32-P4 — read byte → bit[N] for IR module N - bit = 0 → DETECTED (active low — IR sees pill) - bit = 1 → CLEAR

Software read logic

  • pcf8574_read() ส่ง 1-byte read ผ่าน I2C → คืน 8-bit state ของ pin ทั้งหมด
  • Dispense loop poll ที่ ~100 Hz
  • Debounce: 3 LOW samples ติดต่อกัน (30ms) ถึงนับเป็น real pill
  • Block during servo ramp: ระหว่าง pca9685_set_angle_ramped รัน (1.35s) → pcf8574_read() คืน 0xFF (no detection) → กัน phantom triggers จาก EMI servo
  • Error backoff: ถ้า I2C fail ติดต่อกัน → skip 50ms → ลองใหม่

ทำไมต้อง block ระหว่าง ramp?

EMI servo → PCF8574 → IR OUT line wobble
Servo PWM 50Hz กระชากกระแสตอน edge transition → noise spike บนสาย power/ground → PCF8574 internal pull-up เพี้ยน → ดึง IR OUT line ของ IR module ขึ้น/ลงเป็นช่วงสั้นๆ → comparator ของ IR module toggle → LED indicator กระพริบ → PCF อ่านได้เป็น "detected" ผีๆ

Software-side mitigation: ปิดการอ่านระหว่าง ramp transition (จุดที่ noise สูง) → อ่านเฉพาะตอน servo นิ่งที่ WORK position (จุดที่ pill ตกจริง)

Hardware-side (ยังไม่ทำ): ใส่ 100nF cap คร่อม PCF8574 VCC↔GND + bulk 10µF จะลด noise ที่ source ได้มากกว่า

Servo-trust mode (IR ปิด)

ถ้า user ตั้งค่า IR ของ slot นั้นเป็น disabled (ผ่าน /tech/ir endpoint) → dispense loop จะ:

  • ไม่อ่าน PCF8574 (skip IR poll)
  • servo cycle 1 รอบ = นับเป็น 1 pill (trust ระบบกลไก)
  • "Return all" cap loop ที่ stock_count ของ shadow แทน 16

Scheduler (การจัดเวลาจ่ายยา)

Schedule source

เวลาแต่ละ slot ดึงจาก NETPIE MQTT shadow → cache ใน NVS เผื่อ offline

shadow.slot_time[0..6] = ["08:00", "08:30", "12:00", "12:30", "17:00", "17:30", "21:00"]
shadow.med[0..5] = {
  name: "Paracetamol",
  count: 5,
  slots: 0b0000011  // bit 0+1 = slot 0,1 (morning pre+post)
}
shadow.enabled = true

Trigger logic (per-slot fire guard)

  • scheduler ตื่นทุก ~10s
  • คำนวณ delta = cur_minute - slot_minute (fold รอบ midnight)
  • Fire ถ้า 0 ≤ delta ≤ 5 นาที AND s_slot_last_fire[slot] ไม่เคยยิงใน 12 ชม
  • หลังยิง → set s_slot_last_fire[slot] = time(NULL)
ทำไมต้องมี grace 5 นาที?
ถ้า scheduler task ติด dispense ที่ slot ก่อนหน้า อาจตื่นช้าไป 1-2 นาที พอตื่นแล้ว slot ใหม่อาจเลยเวลาไป 1 นาที — strict "diff == 0" เดิมทำให้พลาด. Grace 5 นาทีเก็บความล่าช้าได้ + 12-ชม refire guard กันยิงซ้ำเพราะ grace กว้าง

Network stack

WiFi STA via ESP-Hosted

ESP32-P4 ไม่มี WiFi ในตัว — ใช้ ESP32-C6 เป็น co-processor ผ่าน SDIO 4-bit 40MHz

Reconnect backoff

  • WiFi disconnect event → wifi_schedule_backoff_reconnect()
  • esp_timer one-shot 5s → call esp_wifi_connect()
  • fixed_timer ใน firmware: ถ้า timer active แล้ว → ไม่ re-arm (กัน stuck pending flag)

SNTP (auto time sync)

  • connect แล้ว → SNTP poll pool.ntp.org
  • sync สำเร็จ → ds3231_set_time() update RTC ด้วย
  • resync ทุก 6 ชม กัน RTC drift

NETPIE 2020 MQTT

  • broker mqtt.netpie.io:1883
  • topic @shadow/data/update = receive schedule + med config
  • topic @shadow/data/get = request current shadow
  • parse_shadow JSON → update internal s_shadow → publish to double-buffered snapshot
  • auto reconnect บน event_disconnect

Telegram bot

  • poll getUpdates ทุก 3s
  • commands: /photo, /dispense N M, /return N M, /status, /report YYYY-MM
  • send: text (sendMessage), photo (sendPhoto multipart with JPEG)
  • HTTP lock — กั้น concurrent HTTP client
  • photo inflight gate: max 1 photo upload at a time (queue ถ้ามี dispatch ซ้อน)

Google Sheets logging

  • POST JSON ไป Google Apps Script URL (ผู้ใช้ deploy เอง)
  • field: event (Dispensed/Skipped/Refilled), meds, detail
  • endpoint dead detection: HTTP 404/405 → set s_gsheet_endpoint_dead = stop retrying, ค้าง queue ไว้รอ reboot

Offline queue design

ทำไมต้องมี

WiFi อาจหลุดเป็นช่วงๆ (router reboot, signal weak). ถ้าจ่ายยาตอน WiFi หลุด ก็ยังต้อง:

  • Update shadow.med.count ที่ NETPIE
  • ส่ง Telegram notification
  • Log Google Sheets event

ทุก action ที่ network dependent → queue ใน RAM + persist NVS → flush เมื่อ network กลับมา

Flush logic

  • offline_watch_task ตื่นทุก 5s

IDF v5.3.2 i2c_master ISR race

Critical context: ESP-IDF v5.3.2 มี known bug ใน i2c_master driver — เมื่อ i2c_master_probe() ลอง probe device ที่ไม่มีอยู่จริง บางครั้ง i2c_ll_read_rxfifo ใน ISR จะ dereference NULL/invalid pointer → store fault → panic

Signature

Core0 Saved PC: 0x4ff068f2 (i2c_master_isr_handler_default+656)
Core1 Saved PC: 0x4ff06bd4 (i2c_master_isr_handler_default+682)
ptr=0x6 in i2c_ll_read_rxfifo

Mitigations ที่ shipped

  1. late_detect_task DISABLED ([main.c]) — task เคย ping TCA ทุก 3s × 60 รอบ. ตอนนี้ถ้า user ต้องการตรวจ peripheral ที่เพิ่งต่อใหม่ → ต้อง reboot
  2. i2c_watchdog gated on TCA presence — task probe TCA ทุก 15s; ตอนนี้รันเฉพาะถ้า TCA เจอตอน boot
  3. g_tca_init_ok cache flag — ไม่ re-ping TCA หลัง init เพราะ spurious OK จาก noise ทำให้ VL53 bootstrap (45s I2C lockout) → panic
  4. recover_bus ไม่ esp_restart() — เดิมถ้า mutex timeout → restart → boot loop. ตอนนี้ return error ปกติ + main มี fallback path
  5. i2c_master state-reset retry ใน custom patched components/esp_driver_i2c/ — retry on INVALID_STATE
  6. safe_mode / ultra_safe_mode REMOVED — logic recovery หลัง panic เดิมไม่มีประโยชน์เพราะอุดสาเหตุได้ตรงจุดแล้ว

Root fix (ยังไม่ได้ทำ)

upgrade ESP-IDF เป็น v5.3.3+ หรือเปลี่ยนไปใช้ legacy driver/i2c.h — ทั้งสองตัวเลือกใหญ่ ต้อง regression test ทุก peripheral

เคย: safe_mode / ultra_safe_mode (ลบแล้ว 2026-05-10)

เดิมมี logic เพื่อจัดการ panic recovery — ถ้า boot ก่อนหน้า panic จะเข้า "safe_mode" (ข้าม VL53), ถ้า ≥3 panic ติดกันจะเข้า "ultra_safe_mode" (ข้าม I2C ทั้งหมด). ลบออกเพราะ:

  • Logic ซับซ้อน 18 references ใน main.c กระจาย
  • User เลือกใช้ flow เดียว (online/offline ทำงานเหมือนกัน) ไม่ต้องการ panic-recovery branch
  • Root cause ของ panic (i2c ISR race) ถูกอุดทั้งหมดที่ source แล้ว — ไม่ต้องการ post-panic fallback

หลักคิดสำคัญในการออกแบบ

1. Lazy/early camera init

ปัญหา: Camera SCCB ต้องการ I2C bus ที่บริสุทธิ์ (no stale state) — ถ้า VL53/touch/RTC ใช้ bus อยู่ก่อน → SCCB fail
วิธีแก้: Init camera "early" ก่อน peripheral อื่น ในขณะ I2C bus เพิ่ง init ใหม่ → SCCB ผ่าน → ปิด VL53 XSHUT ระหว่าง SCCB → restore หลังเสร็จ

2. Background camera retry

ปัญหา: ~20-40% ของ cold boot OV5647 ไม่ตอบ SCCB ในรอบแรก (DPHY ยังไม่นิ่ง)
วิธีแก้: spawn cam_retry_task ทุก 8s × 38 ครั้ง (5 นาที total) — usually success ภายใน 1-3 retries

3. Servo ramp (soft motion)

ปัญหา: servo snap ทันที → mechanical jolt → IR sensors สั่น → false detection + EMI กระชากแรง
วิธีแก้: ramp 2°/30ms × N steps → 90° throw ใช้ ~1.35s → smooth motion → ลด jolt + กระจาย current spike ออกไป → IR/I2C นิ่งขึ้น

4. PCF block during ramp

ปัญหา: servo PWM EMI couple เข้า PCF8574 → IR OUT line wobble → phantom triggers
วิธีแก้: pcf8574_block_during_servo_ramp(true) ระหว่าง ramp — pcf8574_read คืน 0xFF (no detection) โดยไม่ไป I2C bus → debounce counter ไม่ latch noise → อ่านจริงเฉพาะตอน servo นิ่ง

5. Strict IR mode (default)

หลักการ: IR เป็น authoritative counter — servo cycle ที่ IR ไม่เห็นยา = นับ 0 pill
เหตุผล: User ต้องการ "ใช้นับยาจริง" ไม่ใช่ trust กลไก (เพราะยาอาจติด ระดับน้ำมัน ฯลฯ)
Servo-trust mode เป็น opt-in เฉพาะ slot ที่ user disable IR ผ่าน UI

6. RTC seed at boot

ปัญหา: System time = epoch 0 หลัง boot → display แสดง "1970-01-01" ก่อน SNTP มา (อาจ 15-30s)
วิธีแก้: ds3231_seed_system_time() ทันทีหลัง I2C ready (t≈2s) → display แสดงเวลาถูกต้องตั้งแต่ first render → SNTP ภายหลังจะ re-sync ทั้ง system time + RTC อีกที

7. Double-buffered shadow snapshot

ปัญหา: parse_shadow update s_shadow fields ทีละตัว → reader ที่อ่านระหว่างนั้นเห็น half-updated data
วิธีแก้: 2 buffer + atomic pointer swap (32-bit aligned = atomic on RISC-V) → reader เห็น snapshot สมบูรณ์เสมอ ถึงแม้ parse กำลังรัน

8. Per-slot last-fire epoch + grace window

ปัญหา: strict diff == 0 minute match → ถ้า scheduler task ติด busy ในช่วง slot นั้น → พลาดทั้ง slot
วิธีแก้: grace 5 min + per-slot 12h refire guard — รับ delay ได้แต่ไม่ยิงซ้ำ

9. Endpoint dead detection (GSheet 405)

ปัญหา: Google Apps Script URL ผิด/redeploy แล้ว → ทุก POST ได้ 405 → retry ทุก 5s → spam log + waste bandwidth
วิธีแก้: 404/405 = config error → set dead flag → stop retrying → log ครั้งเดียวพร้อมคำแนะนำ → events ค้างใน queue (รอด reboot) → user แก้ endpoint + reboot → flush ครั้งเดียว

10. Software block ก่อน hardware fix

หลักคิด: หลายปัญหา (servo EMI, I2C race, NVS limit) มี hardware root cause แต่ user ไม่สะดวกเปลี่ยน hardware ทันที → ทำ software workaround ก่อน → user มีเวลาแก้ hardware ภายหลัง
ข้อแม้: software workaround ต้องไม่ปิดบัง bug — log ชัดเจน + ระบุว่าเป็น workaround

Bug: Camera ค้างที่ boot — backup firmware

Symptom

Board boot → camera init fail × 9-16 ครั้ง → mutex deadlock → SW_CPU_RESET → boot loop ทุก ~50s

Diagnosis

  1. Capture serial log → เห็น recover_bus: cannot take mutex (deadlocked) — restarting
  2. Camera SCCB fail = OV5647 ไม่ตอบ read pid_h
  3. หลายครั้งของ recover_bus → mutex contention → restart

Fix

  • i2c_manager.c:recover_bus — เปลี่ยน 2s timeout → 15s + ลบ esp_restart() → return ESP_ERR_TIMEOUT แทน
  • Background camera retry task มี mutex-busy skip → ไม่ contend กับ recover_bus

Bug: Dead I2C bus (SDA=0 SCL=0)

Symptom

  • I2C scan เจอทุก address (0x09-0x6C = 97 ตัว) → ไม่ใช่ devices จริง
  • recover_bus log Bus still stuck after 4 recovery rounds (SDA=0 SCL=0)
  • ทุก peripheral ขึ้น "OK" แต่อ่านได้ garbage

Root cause

SDA / SCL ค้าง LOW — ไม่มี/อ่อน pull-up resistor, หรือ peripheral พังดึง bus ลง

Fix

  • Hardware: ใส่ pull-up R 4.7kΩ-10kΩ บน SDA, SCL → 3.3V (user ลอง 2.2kΩ ใช้ได้แต่หนัก, ภายหลังถอดออกเพราะ module มี internal pull-up เพียงพอ)
  • การปลด/เสียบ peripheral ระหว่าง debug บางครั้งช่วยปลดล็อก stuck device

Bug: TCA absent → panic loop ทุก ~50s

Symptom

หลังต่อ peripheral ครบ (camera + display + touch + servo + IR + RTC) แต่ไม่ต่อ TCA9548A — boot สำเร็จ แต่ panic ที่ t=50s หลัง init complete

Diagnosis

  • Extract coredump → crashed task = i2c_late, function = i2c_master_probe address 0x70 (TCA)
  • late_detect_task ping TCA ทุก 3s → hit IDF v5.3.2 ISR race → panic
  • หลัง fix late_detect → ยังมี panic — coredump ครั้งใหม่ = i2c_wdog task ping TCA ทุก 15s

Fix

  1. Disable late_detect_task spawn เลย — TCA9548A เป็น soldered IC ไม่ hot-plug
  2. Gate i2c_watchdog_task on g_tca_init_ok — ไม่สร้าง task ถ้า TCA absent
  3. Cache TCA init result เป็น g_tca_init_ok — ไม่ re-ping (spurious OK จะทำให้ VL53 bootstrap)

Bug: Telegram /photo คืนภาพดำ (E2 revert)

Symptom

หลัง flash overnight patches → /photo ส่งภาพ JPEG ที่เปิดแล้วเป็นสีดำทั้งหมด — เดิมทำงานปกติ

Diagnosis

  • Camera MIPI ยัง alive (frame counter ขึ้นปกติ 5500+ frames)
  • JPEG encoder ทำงาน — Telegram receive content length = 1197 (response, not request)
  • ภาพถูกส่งสำเร็จแต่ pixel data เป็น 0 → encoder อาจอ่าน buffer ที่ยังไม่ถูก write

Root cause

Audit ของ overnight patches เสนอ E2 fix: ใส่ portENTER_CRITICAL_ISR / portENTER_CRITICAL ใน camera ISR handlers + encoder task — ป้องกัน "race" ระหว่าง buffer index reads/writes.
แต่ critical sections เพิ่ม latency เล็กน้อยใน ISR — ทำให้ timing ระหว่าง DMA-finished + encoder-pickup เลื่อนนิดเดียว → encoder pick buffer ก่อน DMA flush ลง PSRAM → encode raw memory ว่าง

Fix

Revert E2 — กลับใช้ volatile int + atomic 32-bit read/write (ก่อนหน้านี้ทำงานหลายเดือนไม่มี race จริงๆ) → ภาพกลับมาปกติ

บทเรียน: Audit's "Critical" severity ไม่ได้แปลว่าโค้ดเดิม buggy — บางครั้ง "theoretical race" ที่ตามทฤษฎีน่าจะมี practical impact 0 บนสถาปัตยกรรมจริง (RISC-V 32-bit aligned int = atomic read/write). Live test เท่านั้นที่จะรู้

งานที่ค้าง / ต่อยอด

Hardware (สำคัญ)

  • ใส่ 100nF cap คร่อม PCF8574 VCC↔GND ที่ตัว chip — ลด EMI coupling ที่ source (ปัจจุบัน software block แค่ mitigate)
  • ตรวจ pull-up เพิ่ม ถ้า bus เพี้ยนหลังต่อ peripheral ครบ
  • ปรับ IR pot ทีละตัว ให้ LED ดับสนิทเวลาไม่มียา
  • (ทางเลือก) ย้าย IR จาก PCF8574 ไป GPIO ตรง ใช้ GPIO 2-6, 46 — ตัด PCF ออกจาก path = ตัด noise คัปเพิล

Software (Medium)

  • J1 — jpeg_enc release frame watchdog: ถ้า caller ไม่ release ใน N วินาที → auto-release (กัน buffer pool leak)
  • J2 — DMA2D timeout tune based on PSRAM load
  • M2 — pca9685 servo angle atomicity ตอน ramp กับ web /servo/set
  • Per-key NVS storage สำหรับ offline_sync — ลบ 4KB blob limit
  • Wakeup จาก deep sleep ตอน scheduled slot ใกล้ถึง (ประหยัดไฟ)

IDF

  • Upgrade ESP-IDF เป็น v5.3.3+ หรือ v5.4 → root fix ของ i2c_master ISR race
  • หลังอัพ → ลบ components/esp_driver_i2c/ custom patch ออก
